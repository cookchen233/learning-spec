# Concept Model

这个文件定义 `learning-spec` 的核心概念边界。

## 1. 总原则

`learning-spec` 的目标不是把学习内容切得越细越好，而是把长期学习目标稳定转化为：

- 可被长期约束的现实边界
- 可被感知的能力获得单元
- 可被奖励系统感知的阶段跃迁
- 可被持续恢复的执行状态

它优先按能力的物理结构拆解，而不是按自我设定的期限拆解。

它同时承认一个现实：

- 人类不适合提前规划很远的即时动作
- 人类更适合提前规划 `level / zone / capability`
- 即时动作应在临近学习时生成
- 期限只是一种现实约束，不是正式结构的主切分依据

## 2. 核心对象

### Project Contract

`project-contract` 是长期现实契约层。

它回答：

- 这项学习的长期目标是什么
- 每天现实可投入资源是什么
- 哪些长期规则不能被局部波动轻易改变

### Level

`level` 是阶段跃迁单元。

它回答：

- 当若干关键能力成立后，我整体处在什么阶段
- 我已经不再是哪一类初学者

`level` 不是学习内容清单，而是身份与能力层级的变化。

### Zone

`zone` 是奖励化领域单元。

它回答：

- 我拿下了哪一块现实沟通领域
- 一组同类 `capability` 共同构成了哪块“地盘”

`zone` 既有领域组织作用，也应有明显获得感。

### Capability

`capability` 是最小可验证能力单元。

它回答：

- 我稳定获得了什么能力
- 下次遇到类似情境时，我是否仍能完成

`capability` 不是“一次练习完成”，而是“可重复调用的能力成立”。

一个正式 `capability` 默认应拥有自己的目录。

当它进入执行期时，可以在该目录下生成近程 `task-*.md`。

### Task

`task` 是近程执行单元。

它回答：

- 围绕当前 capability，这一次学习窗口要做什么
- 本次动作如何一步步把 capability 推向成立

在 `learning-spec` 中：

- `task` 存在
- 但只在当前 capability 被执行时生成
- 不提前为远期 capability 铺开大量 task

### Execution State

`execution-state` 是连续性状态账本。

它回答：

- 当前 `level` 内推进到哪个 `zone / capability / task`
- 当前真实情况是什么
- 下一次学习窗口开始时，应立即做什么

默认情况下，一个 `execution-state` 对应一个当前激活的 `level`，并放在该 `level` 目录下。

## 3. 关系结构

```text
Project Contract
  -> 提供长期现实约束

Level
  -> 表示阶段跃迁

Zone
  -> 表示某块领域被拿下

Capability
  -> 表示最小能力成立

Task
  -> 表示围绕当前 capability 的近程执行动作

Execution State
  -> 记录当前 level 内的位置与即时 session plan
```

## 4. 边界

- `level` 不是奖励口号，必须对应真实能力阶段变化
- `zone` 不是学术分类词，必须让人感到一块领域被拿下
- `capability` 不是抽象能力标签，必须可验证、可复现
- `task` 不是远期规划层，只是当前 capability 的近程执行层

## 5. 能力成立

一个 `capability` 成立，不是因为：

- 接触过这个主题
- 做过一次练习
- 背过相关表达

而是因为：

- 在对应场景里可以稳定完成
- 至少有 5 个可复现例子支撑
- 这些例子中每个例子都至少包含 3 个可替换元素
- 不同例子之间不能只是机械替换单个词
- 可以被录音、复述、模拟对话或真实交流证明
- 下次遇到类似情境时仍可调用
