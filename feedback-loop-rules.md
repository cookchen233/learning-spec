# Feedback Loop Rules

`learning-spec` 必须通过真实学习实践持续修正。

## 哪些问题必须记录

- capability 明显过大，长期无法成立
- zone 虽然结构整齐，但没有获得感
- level 完成后没有真实跃迁感
- 证据要求太弱，导致伪进展
- 证据要求太重，导致系统无法持续
- session plan 总是失控，说明 capability 定义可能不对

## 记录位置

默认记录到 `execution-state` 的 `Practice Feedback`。

## 反馈字段

- `Observed Problem`
- `Spec Impact`
- `Proposed Fix`
- `Status`

## 闭环要求

1. 先记录问题
2. 再判断是实例问题还是系统问题
3. 若属于系统问题，回写模板或规则
4. 更新状态
